I Pair Program Every Day

And Want To Tell You All About It

Mike Mazur
Saturday, May 5, 2012

Agenda

  • What is pairing?
  • What is my experience with pairing?
  • How do we pair? (Mechanics)
  • Why do we pair? (Pros)
  • Pitfalls & challenges (Cons)
  • Conclusions
  • Side note: Interruptions and questions welcome!

What is Pairing?

  • Two people working on the same problem
  • One driver, one navigator
  • One low level (i.e.: syntax), one high level (strategy, integration, thinking ahead)
  • Who’s done pair programming before?

My Experience

  • Pairing full time since July 2010 at Pivotal Labs/New Context
  • We pair on all production code
  • Complicated "tricky" code
  • Simple CSS fixes
  • Ops even

How Does it Actually Work?

  • 2 keyboards, 2 mice, 27" & 24" screens
  • Standard workstation image and tooling
  • Rotate pairs every day
  • No one has their own workstation
  • Everyone can and does use pretty much any machine
  • Email stations available for personal tasks
  • Remote pairing with Skype, iChat

Why?

  • Knowledge sharing empowers each developer to contribute because knowledge & process spread
  • Quality code delivered faster & cheaper, for product owner to see results & iterate
  • Bonus: Great way to interview candidates

Empowered Team, Sustainable Project

  • Knowledge sharing (tech, design approaches & decisions)
    • through pair rotation
    • learn new tech, best practices, readable code
    • rotation among components breaks down silos & bottlenecks
    • diversify skill set (i.e.: DBA, frontend)
  • Teach them entire / whole process (TDD, CI, product ppl)
  • Project history
  • Learn fast!

Code Quality, Less Bugs

  • Always two viewpoints, backgrounds, experiences
    • deeply immersed
  • Who does code review?
  • Constant code review
  • Bugs are often caught as they’re introduced
    • tables, secure cookie, slightly better understanding (new vs. create)
    • pressure

Cost Savings

  • Decrease impact of interruptions (huge!)
    • both external & self-inflicted
    • one person handles interrupt, other stays focused
  • Focus on business value, not interesting engineering problem to solve
  • Clearer understanding of requirements (having product on-site really helps)
  • Clear code, easier to maintain in the long run
  • Important issues thrashed out on the spot
  • Naming
    • "What should we call this exception?"
    • clarity, simplicity

Challenges

  • Difficult to keep own hours or work from home
    • an adjustment for me
  • Being focused 8 hours per day can be draining
    • but is fun & fulfilling
    • you get used to it
  • Research situations…

Difficult Pairs

  • Distracted people (iPad, phone, email)
    • feels like you’re soloing
  • No passion, heart is not in it
    • the person makes no progress in developing their skills
    • minimal contribution
  • Crowders

Summary

  • Pairing is awesome, highly recommended
  • Learning pace outstanding, I made huge progress
  • Can be addictive churning out high-quality product
